# Нефункциональные требования

## Доступность

API Gateway, Order Service, Payment Service и Robot Service являются критичными компонентами. Для них требуется горизонтальное масштабирование и health-check'и.

## Масштабируемость

Клиентские приложения и API Gateway масштабируются независимо от бизнес-сервисов. Order Service и Menu Service могут масштабироваться отдельно, так как имеют разные профили нагрузки.

## Согласованность данных

Система использует eventual consistency для сценариев оплаты и приготовления. Статус заказа меняется по мере получения событий от Payment Service и Robot Service.

## Безопасность

Авторизация выполняется через Auth Service. API Gateway проверяет токены и не пропускает неавторизованные запросы во внутренние сервисы.

## Наблюдаемость

Для production-версии требуются:

- централизованные логи;
- distributed tracing;
- метрики Kafka consumer lag;
- алерты по ошибкам оплаты и приготовления;
- business metrics по заказам и платежам.

## Отказоустойчивость

Асинхронные взаимодействия через Kafka снижают связанность сервисов, но требуют идемпотентных consumer'ов и обработки повторной доставки сообщений.
