# ADR-0001. Использовать микросервисную архитектуру

## Статус

Принято.

## Контекст

Food Ordering Platform включает несколько бизнес-возможностей: заказы, меню, ингредиенты, авторизация, оплата и приготовление. Эти области имеют разные причины для изменения и разные требования к масштабированию.

## Решение

Разделить систему на микросервисы по бизнес-возможностям:

- Order Service;
- Menu Service;
- Ingredient Service;
- Auth Service;
- Payment Service;
- Robot Service.

## Последствия

Положительные последствия:

- сервисы можно развивать независимо;
- каждый сервис владеет своими данными;
- упрощается масштабирование отдельных частей системы.

Отрицательные последствия:

- появляются распределённые бизнес-процессы;
- требуется наблюдаемость;
- требуется управление eventual consistency.
